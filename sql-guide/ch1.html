Chapter 1: Getting Started

Installation, Tools, and Your First Queries

1.1 Why SQL for Analytics?

SQL (Structured Query Language) is the universal language for working with relational databases. Whether you are pulling sales reports, analyzing customer behavior, or building dashboards, SQL is the tool that sits between you and the data.

Almost every company stores its operational data in a relational database. Orders, customers, products, inventory levels, employee records: all of these live in tables made up of rows and columns. SQL lets you ask precise questions of that data without writing a full program.

The relational model organizes data into tables (also called relations). Each table has a fixed set of columns (attributes) and a variable number of rows (records). A column in one table can reference a column in another through a foreign key, linking the tables together. This structure eliminates redundancy and keeps data consistent.

Concept: Declarative vs. Procedural

SQL is declarative: you describe what data you want, not how to retrieve it. The database engine decides the most efficient execution plan. This is different from procedural languages like Python, where you write step-by-step instructions.

1.2 Installing SQLite

SQLite is the simplest way to start. It requires no server, no configuration, and runs as a single file on your computer. Download it from sqlite.org.

macOS: SQLite comes pre-installed. Open Terminal and type:

-- Check your SQLite version
sqlite3 --version

Windows: Download the "sqlite-tools" zip, extract it, and add the folder to your system PATH. Then open Command Prompt:

sqlite3 my_first_database.db

You should see the SQLite prompt: sqlite>. Type .quit to exit.

1.3 Database Tools

While the command line works, a graphical tool makes learning easier. Here are the most popular options:

  • DB Browser for SQLite (free, open-source) — lightweight, great for beginners working with SQLite files.
  • DBeaver (free Community Edition) — supports dozens of databases; good once you move beyond SQLite.
  • DataGrip (JetBrains, paid) — full-featured IDE with intelligent code completion and refactoring.
  • pgAdmin (free) — the standard tool for PostgreSQL administration and querying.

For this guide, any of these tools will work. Pick one and connect it to a local SQLite file to follow along.

1.4 Creating Your First Database

A database starts with tables. The CREATE TABLE statement defines a table's name and columns. Each column has a name and a data type.

CREATE TABLE customers (
    customer_id   INTEGER PRIMARY KEY,
    first_name    TEXT NOT NULL,
    last_name     TEXT NOT NULL,
    email         TEXT,
    signup_date   TEXT    -- SQLite stores dates as TEXT
);
CREATE TABLE products (
    product_id    INTEGER PRIMARY KEY,
    product_name  TEXT NOT NULL,
    category      TEXT,
    unit_price    REAL,
    stock_qty     INTEGER DEFAULT 0
);
CREATE TABLE orders (
    order_id      INTEGER PRIMARY KEY,
    customer_id   INTEGER REFERENCES customers(customer_id),
    order_date    TEXT,
    total_amount  REAL
);

Tip: Common Data Types

INTEGER — whole numbers (IDs, quantities). TEXT — variable-length strings (names, emails). REAL — floating-point numbers (prices, measurements). In PostgreSQL you also have DATE, TIMESTAMP, NUMERIC, and BOOLEAN.

1.6 Your First SELECT

The SELECT statement retrieves data. The simplest form selects all columns from a table.

-- Select all columns and all rows
SELECT * FROM customers;

In practice, you should name the columns you need rather than using *. This makes your query self-documenting and avoids pulling unnecessary data.

-- Select specific columns
SELECT first_name, last_name, email
FROM customers;

You can rename columns in the output using AS (an alias):

SELECT
    product_name        AS product,
    unit_price          AS price,
    unit_price * 1.08  AS price_with_tax,
    stock_qty           AS inventory
FROM products;

Notice the computed column price_with_tax. SQL lets you write expressions directly in the SELECT list. The result is calculated on the fly and does not change the stored data.

-- Quick revenue estimate per product
SELECT
    product_name,
    unit_price,
    stock_qty,
    unit_price * stock_qty AS inventory_value
FROM products;

1.7 SQL Dialects

SQL has an ISO standard, but every database adds its own extensions. The core statements (SELECT, INSERT, UPDATE, DELETE) work the same way everywhere. Differences show up in data types, functions, and advanced features.

This guide uses SQLite syntax. When a concept differs in other databases, we will note it explicitly.

Warning: Portability Pitfalls

If you write queries in SQLite and later move to PostgreSQL, watch out for: (1) date/time functions are completely different, (2) SQLite has dynamic typing while PostgreSQL enforces types strictly, and (3) SQLite's GROUP BY allows non-aggregated columns that other databases reject.

1.8 Comments and Formatting

Good formatting makes SQL readable. Use comments to explain your logic, especially in long queries.

-- Single-line comment: everything after -- is ignored
SELECT first_name, last_name
FROM customers;

/*
   Multi-line comment:
   This query calculates inventory value
   for products in the Electronics category.
*/
SELECT
    product_name,
    unit_price * stock_qty AS inventory_value
FROM products
WHERE category = 'Electronics';

Formatting conventions:

  • Write SQL keywords in UPPERCASE (SELECT, FROM, WHERE).
  • Write table and column names in lowercase with underscores (order_date, customer_id).
  • Put each major clause on its own line.
  • Indent column lists and conditions for readability.
  • End every statement with a semicolon.

1.9 Common Beginner Errors

Every SQL learner hits these mistakes. Recognizing them saves hours of frustration.

Missing semicolon: Most SQL tools expect a semicolon to mark the end of a statement. Without it, the tool waits for more input.

-- This will hang in many tools (no semicolon):
SELECT * FROM customers

-- Fix: always end with a semicolon
SELECT * FROM customers;

Case sensitivity: SQL keywords are case-insensitive (SELECT and select both work). However, string comparisons may be case-sensitive depending on the database. In SQLite, LIKE is case-insensitive for ASCII characters, but = is case-sensitive.

-- These return different results in SQLite:
SELECT * FROM customers WHERE first_name = 'alice';   -- no match
SELECT * FROM customers WHERE first_name = 'Alice';   -- match

Quoting mistakes: Use single quotes for string values. Double quotes are for identifiers (column/table names) in standard SQL. Mixing them up causes errors.

-- Correct: single quotes for values
SELECT * FROM products WHERE category = 'Electronics';

-- Wrong: double quotes treat it as a column name
-- SELECT * FROM products WHERE category = "Electronics";
-- (This works in SQLite/MySQL but fails in PostgreSQL)

Tip: Debugging Strategy

When a query fails, try three things: (1) check for missing commas between column names, (2) check for missing or extra parentheses, and (3) run just the FROM clause with SELECT * to make sure the table exists and has the columns you expect.

Key Takeaways

  • SQL is a declarative language for querying relational databases. You describe what you want; the engine figures out how.
  • SQLite is the easiest database to start with: no server, no setup, just a single file.
  • CREATE TABLE defines structure; INSERT INTO adds data; SELECT retrieves data.
  • Always name your columns explicitly instead of using SELECT * in production queries.
  • Use AS to give readable aliases to columns and computed expressions.
  • SQL dialects share core syntax but differ in functions, data types, and edge cases. Write portable SQL when you can.
  • Format your queries consistently: uppercase keywords, lowercase identifiers, one clause per line, and always end with a semicolon.

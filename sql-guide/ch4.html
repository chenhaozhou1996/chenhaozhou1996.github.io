Chapter 4: Aggregation & GROUP BY

Summarizing Data

Raw detail rows are useful, but business decisions require summaries. How much revenue did we earn last quarter? Which product category sells best? Aggregation functions collapse many rows into a single summary value, and GROUP BY lets you produce those summaries for each group independently.

4.1 Aggregate Functions

SQL provides five core aggregate functions. Each takes a set of values from multiple rows and returns one result.

-- Basic aggregate functions on the orders table
SELECT
    COUNT(*)        AS total_orders,
    SUM(amount)     AS total_revenue,
    AVG(amount)     AS avg_order_value,
    MIN(amount)     AS smallest_order,
    MAX(amount)     AS largest_order
FROM orders
WHERE order_date >= '2024-01-01';

Aggregates Collapse Rows

When you use an aggregate function without GROUP BY, the entire result set is treated as one group. The query above returns exactly one row, no matter how many orders exist.

-- Combining aggregates with a filter
SELECT
    COUNT(*)                           AS num_products,
    AVG(unit_price)                    AS avg_price,
    MAX(unit_price) - MIN(unit_price)  AS price_range
FROM products
WHERE discontinued = 0;

4.2 COUNT Variations

COUNT behaves differently depending on its argument. Understanding the three forms prevents subtle bugs.

SELECT
    COUNT(*)                 AS total_rows,
    COUNT(ship_date)         AS shipped_orders,
    COUNT(DISTINCT customer_id) AS unique_customers
FROM orders;

COUNT(*) vs COUNT(column)

COUNT(*) counts every row, including those with NULLs. COUNT(column) counts only non-NULL values in that column. COUNT(DISTINCT column) counts unique non-NULL values. If ship_date is NULL for unshipped orders, COUNT(ship_date) excludes them.

4.3 GROUP BY Basics

GROUP BY splits the result set into groups. Each group gets its own aggregate calculation.

-- Revenue by product category
SELECT
    category,
    COUNT(*)       AS num_orders,
    SUM(amount)    AS total_revenue,
    AVG(amount)    AS avg_order_value
FROM orders o
JOIN products p ON o.product_id = p.product_id
GROUP BY category
ORDER BY total_revenue DESC;
-- Grouping by multiple columns: sales by region and quarter
SELECT
    region,
    EXTRACT(YEAR FROM order_date)    AS order_year,
    EXTRACT(QUARTER FROM order_date) AS order_quarter,
    SUM(amount)                        AS quarterly_revenue,
    COUNT(DISTINCT customer_id)        AS active_customers
FROM orders
GROUP BY region,
         EXTRACT(YEAR FROM order_date),
         EXTRACT(QUARTER FROM order_date)
ORDER BY region, order_year, order_quarter;

Every non-aggregated column in your SELECT list must appear in the GROUP BY clause. Violating this rule produces an error in standard SQL (though MySQL historically allowed it with unpredictable results).

4.4 HAVING Clause

WHERE filters individual rows before grouping. HAVING filters groups after aggregation. This distinction matters because you cannot reference aggregate results in a WHERE clause.

-- Find categories with more than $50,000 in total sales
SELECT
    category,
    SUM(amount) AS total_sales,
    COUNT(*)    AS order_count
FROM orders o
JOIN products p ON o.product_id = p.product_id
WHERE order_date >= '2024-01-01'      -- row-level filter
GROUP BY category
HAVING SUM(amount) > 50000             -- group-level filter
ORDER BY total_sales DESC;
-- Customers who placed at least 5 orders with average value above $200
SELECT
    customer_id,
    COUNT(*)    AS order_count,
    AVG(amount) AS avg_order_value
FROM orders
GROUP BY customer_id
HAVING COUNT(*) >= 5
   AND AVG(amount) > 200;

Execution Order Matters

The logical processing order is: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. WHERE cannot see aggregate results because aggregation has not happened yet. HAVING operates on the already-formed groups.

4.5 Aggregation with NULL

All aggregate functions except COUNT(*) ignore NULL values. This can produce surprising results if you are not careful.

-- Suppose discount is NULL when no discount was applied
SELECT
    COUNT(*)           AS total_orders,
    COUNT(discount)    AS discounted_orders,
    AVG(discount)      AS avg_discount_given,
    AVG(COALESCE(discount, 0)) AS avg_discount_all
FROM order_details;

In this example, AVG(discount) averages only the rows that actually have a discount, yielding a higher number. AVG(COALESCE(discount, 0)) treats missing discounts as zero, giving the true average across all orders.

-- SUM of an all-NULL group returns NULL, not zero
SELECT
    department,
    COALESCE(SUM(bonus), 0) AS total_bonus
FROM employees
GROUP BY department;

4.6 GROUP BY with Expressions

You can group by computed expressions, not just raw columns. This is especially useful for date-based summaries and conditional aggregation.

-- Monthly revenue summary
SELECT
    DATE_TRUNC('month', order_date) AS month,
    SUM(amount)                       AS monthly_revenue,
    COUNT(DISTINCT customer_id)       AS unique_buyers
FROM orders
GROUP BY DATE_TRUNC('month', order_date)
ORDER BY month;
-- Conditional aggregation with CASE inside SUM
SELECT
    region,
    COUNT(*) AS total_orders,
    SUM(CASE WHEN status = 'completed'  THEN 1 ELSE 0 END) AS completed,
    SUM(CASE WHEN status = 'cancelled'  THEN 1 ELSE 0 END) AS cancelled,
    ROUND(
        100.0 * SUM(CASE WHEN status = 'cancelled' THEN 1 ELSE 0 END)
        / COUNT(*), 1
    ) AS cancel_rate_pct
FROM orders
GROUP BY region;

Conditional aggregation with CASE is a powerful pattern. It lets you pivot data and compute multiple filtered metrics in a single pass, avoiding the need for repeated subqueries.

4.7 ROLLUP and CUBE

Standard GROUP BY produces one level of grouping. ROLLUP adds subtotals and a grand total. CUBE generates all possible subtotal combinations. Both are supported in PostgreSQL, SQL Server, and Oracle.

-- ROLLUP: subtotals by region, then grand total
SELECT
    COALESCE(region, 'ALL REGIONS')   AS region,
    COALESCE(category, 'ALL CATEGORIES') AS category,
    SUM(amount)   AS total_sales,
    COUNT(*)      AS num_orders
FROM orders o
JOIN products p ON o.product_id = p.product_id
GROUP BY ROLLUP(region, category)
ORDER BY region NULLS LAST, category NULLS LAST;
-- CUBE: subtotals for every combination of region and year
SELECT
    region,
    EXTRACT(YEAR FROM order_date) AS yr,
    SUM(amount) AS revenue
FROM orders
GROUP BY CUBE(region, EXTRACT(YEAR FROM order_date))
ORDER BY region NULLS LAST, yr NULLS LAST;

ROLLUP(A, B) produces grouping sets: (A, B), (A), and (). CUBE(A, B) produces: (A, B), (A), (B), and (). Use GROUPING() to distinguish real NULLs from subtotal NULLs when needed.

4.8 STRING_AGG / GROUP_CONCAT

Sometimes you need to concatenate text values within a group rather than count or sum them.

-- PostgreSQL: list all product names per category
SELECT
    category,
    COUNT(*) AS num_products,
    STRING_AGG(product_name, ', ' ORDER BY product_name) AS product_list
FROM products
GROUP BY category;
-- MySQL equivalent using GROUP_CONCAT
SELECT
    department,
    GROUP_CONCAT(employee_name ORDER BY hire_date SEPARATOR ', ') AS team_members
FROM employees
GROUP BY department;
-- Practical example: customer order history summary
SELECT
    c.customer_name,
    COUNT(o.order_id) AS total_orders,
    SUM(o.amount)     AS lifetime_value,
    MIN(o.order_date) AS first_order,
    MAX(o.order_date) AS last_order,
    STRING_AGG(DISTINCT p.category, ', ' ORDER BY p.category)
        AS categories_purchased
FROM customers c
JOIN orders o    ON c.customer_id = o.customer_id
JOIN products p  ON o.product_id = p.product_id
GROUP BY c.customer_id, c.customer_name
ORDER BY lifetime_value DESC;

Key Takeaways

  • The five core aggregates (COUNT, SUM, AVG, MIN, MAX) collapse rows into summary values.
  • COUNT(*) counts all rows; COUNT(col) skips NULLs; COUNT(DISTINCT col) counts unique non-NULL values.
  • GROUP BY produces one output row per group. Every non-aggregated SELECT column must appear in GROUP BY.
  • WHERE filters rows before grouping; HAVING filters groups after aggregation.
  • All aggregates except COUNT(*) ignore NULLs. Use COALESCE when zero should replace NULL.
  • CASE inside an aggregate enables conditional aggregation, a versatile pattern for pivoting and rate calculations.
  • ROLLUP and CUBE generate subtotals and grand totals automatically.
